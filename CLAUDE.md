# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"EDS — Especialista em Cantos" is a Brazilian-Portuguese, single-page PWA that
projects football (soccer) **corner-kick (escanteio)** totals across 9 active
leagues (BR, BR_B, ARG, ARG_B, MLS, USL, BUN, CHI, ECU — see `_LIGAS_TODAS` in
`index.html`) and turns those projections into betting recommendations
(over/under, Asian handicap, multiplas/parlays). There is no build step, no
bundler, no package manager at the root — the entire app is one static HTML
file plus data files served as-is.

## Running the app locally

```bash
abrir-app.bat        # Windows: starts `python -m http.server 8765` and opens index.html
# or manually:
python -m http.server 8765
```

Open via `http://localhost:8765`, not `file://` — the CSP and the service
worker (offline caching, PWA install) misbehave under `file://`.

There is no lint/test/build command for the main app — it's plain
HTML/CSS/JS with no compilation step. Verify changes by loading the page in a
browser and exercising the relevant view (see Architecture below); check the
browser console for errors.


## Architecture of `index.html`

This is a large, hand-rolled single-file app. There's no framework — plain
DOM manipulation, template strings, and global functions. To work in it
productively, know these layers:

### 1. Data loading
Near the top of the body, a long series of `<script src="data/*.js?v=...">`
tags load each league's dataset by setting a global, e.g.
`window.DADOS_BR`, `window.DADOS_MLS`, etc. (see `_LIGAS_PRIORIDADE` and
`_LIGAS_TODAS` in `index.html`). Each dataset is a plain JS object with
`times` (team list) and `jogos` (matches with per-period corner/shot/
possession stats). These files are **generated by the scraper pipeline**, not
hand-edited — always bump the `?v=YYYY-MM-DD-HHmm` cache-busting query string
in the `<script src>` tags when a dataset changes (the service worker relies
on this + network-first for `/data/`).

Auth gates the whole app: a Supabase client (`window._sbAuth`, public
project `onuqpfawzchrbluvrvfj`) shows a login/signup overlay until a session
exists (see the IIFE right after `<body>`).

### 2. The projection engine
Everything funnels through `projecaoJogo(mandante, visitante)` — this is the
core function that turns two teams into a full projection (expected corners
per period, TMI, Sniper signals, protocol flags, etc). Supporting pieces:

- `MOTOR_CONFIG` — all "magic numbers" for the engine (EWMA decay, Bayesian
  shrinkage k, TMI clamps, Sniper thresholds, confidence formula). **Comments
  above each constant explain the reasoning** — read them before touching a
  value, and bump the engine version comment when recalibrating from a
  backtest.
- `calcDNA`, `avgWeighted`, `shrink`, `poisson`, `getOvers` — statistical
  primitives (EWMA, Bayesian shrinkage toward league mean, Poisson
  probabilities).
- `calcularOARF` / `getForcaOfensiva` / `getForcaDefensiva` — Opponent-
  Adjusted Recent Form, used to avoid over-reacting to a hot/cold streak.
- `getRankingHT`, `calcularForcaCantos` — league-wide rankings (used by the
  stake protocol's "cross-ranking" filter, see below).
- `LIGA_DNA`, `ARQUETIPO_MULT`, `getLigaDNA` — qualitative per-league/
  archetype multipliers ("DNA Escoteiro").
- `gerarRecomendacoes(p)` — turns a projection into ranked bet
  recommendations, applying the stake protocol.

**`PROTOCOLO_STAKE`** (constant + logic spread through `projecaoJogo` and
`gerarRecomendacoes`) encodes hard-won betting rules from real backtests
(documented in full in `backtest/protocolo_stake.md`): e.g. only surface an
HT Asian Handicap pick when the home team is top-5 attack AND the away team
is top-10 defense (a cross-ranking filter), and reinforce Under FT stake when
both teams' TMI < 1.0. **When changing these thresholds, update
`backtest/protocolo_stake.md`'s changelog too** — it's the source of truth
for *why* a rule exists.

`LIGAS_STATUS` is a per-league calibration registry (ok/observação/suspenso)
driven by `backtest/analisar.js` output — update it after each new bet batch
per the loop documented in `backtest/README.md`.

### 3. Views / navigation
The UI is organized as **groups → views**, not routes:
- `VIEW_TO_GRUPO` maps every view id (`dashboard`, `teacher`, `enigma`,
  `tracker`, `auditoria`, ...) to one of 4 top-level groups (`visao`,
  `analise`, `operacao`, `auditoria`).
- `SUBNAV_BY_GRUPO` defines the sub-nav buttons per group.
- `showView(id)` swaps the visible `.view` element and calls that view's
  `render*()` function (`renderDashboard`, `renderRanking`, `renderTeacher`,
  `renderOuro`, `renderAuditoria`, `renderTracker`, ...).

Distinct "signal families" surfaced as separate views/exports, each with its
own render + export function (`exportar*HTML`/`exportar*JSON`): Teacher
(main projection breakdown), Bala de Prata, Cisne Negro, Vencedor Cantos,
Filtro OURO, ENIGMA (territorial-discrepancy scanner), Múltiplas
Elite/Argentina (parlay builder), Pinnacle (real market odds + EV vs the
model).

### 4. Auditoria (cross-league audit trail)
`renderAuditoria` + `_snapshotLigaAtual`/`_classificarLiga` implement a
"memória institucional": each time a matchday's data is loaded, a snapshot
hash is stored (localStorage) so later analyses can detect whether the
underlying data changed since a report was generated. Exports go to
`Auditoria Especialista em cantos/` conventionally (see `AUDITORIA_LIGAS`).

### 5. Bet tracker & Pinnacle CSV import
`renderTracker`, `salvarBetTracker`, `importarCSVPinnacle`,
`calcularCLV*`, `calcularROIComIC`, `detectarDriftWR` — a manual bet log
(localStorage-backed, key from `betsKey()`) with CLV (closing-line value),
ROI with confidence interval, and win-rate drift detection. This is
independent of the `backtest/` directory (which tracks *externally* logged
real-money bets used to recalibrate the engine); the in-app tracker is for
day-to-day bet logging by the user.

### 6. SmartCoach (in-app AI assistant)
A chat panel (`toggleSC`, `addSCMsg`, `getAppContext`, functions prefixed
`_sc*`) calls `https://api.anthropic.com/v1/messages` **directly from the
browser** using a user-supplied API key stored client-side (never commit a
real key). `getAppContext()` assembles the current app state (rankings,
memory, calibration) as context fed to the model. This same
`api.anthropic.com` call pattern is reused for OCR-style bet-slip reading
(around the `capturarImagem`/screenshot flows) — search for
`api.anthropic.com` to find all call sites if updating the model id or
request shape.

### 7. MCP bridge (Supabase)
`_edsSyncRodada` / `_edsSyncMultiplas` fire-and-forget upsert the current
matchday's games + computed signals (Cisne Negro, Enigma, Bala de Prata,
Sniper) and generated multiplas into Supabase tables (`rodadas_diarias`,
`multiplas_geradas`), keyed by the logged-in user. This lets an external MCP
connector query the *real* engine output without reimplementing the engine
in another runtime — never let these calls block or throw into the UI (they
are wrapped in try/catch by design; preserve that when editing).

### CSP
`index.html`'s `<meta http-equiv="Content-Security-Policy">` allowlists exact
external hosts (jsdelivr, cdnjs, supabase, `api.anthropic.com`,
tessdata.projectnaptha.com, unpkg). **Adding any new external script/fetch
target requires updating this CSP**, or it will be silently blocked.

## Scraper pipeline (`scraper/`)

Node + Playwright scripts scrape FlashScore per league and produce the
`data/*.js` files consumed by `index.html`:

1. `minerador_<liga>.js` (or `extrator.js`) — Playwright crawler, visits a
   league's results page on FlashScore, pulls per-match corner/shot/
   possession stats (HT + FT) via `parseFlashscoreStats`, writes
   `raw_<liga>.json`.
2. `formatar_<liga>.js` — maps FlashScore team names to the app's canonical
   names (see the `mapTimes` alias dictionaries — these need updating
   whenever FlashScore renames a club or a new club is promoted) and emits
   the final `window.DADOS_<LIGA> = {...}` file into `data/`.
3. One-off `_*.js` scripts (`_INJETAR_RODADA_*`, `_AUDITAR_*`,
   `_ANALISAR_*`) are ad-hoc audit/injection scripts for specific matchdays
   — not part of a stable pipeline, treat as disposable/historical.

Run with `node scraper/<script>.js` (only `playwright` is a dependency, see
`scraper/package.json`). Debug artifacts (`raw_*.json`,
`diagnose_html.txt`, etc.) are gitignored — never commit them.

## Backtest / calibration loop

`backtest/` holds real-money Pinnacle bet logs (`bets_*.json`) and
`analisar.js`, which computes win rate, ROI, and edge-vs-market per league
and per market type. This is how `MOTOR_CONFIG`'s `OARF_PESO_POR_LIGA` and
`LIGAS_STATUS` get recalibrated — **never tune those weights from intuition;
follow the loop in `backtest/README.md`** (collect ≥15 bets/league, run
`analisar.js`, only adjust after 2+ consecutive batches confirm the trend,
commit the config change separately from the data collection, and reference
the batch that justified it).

```bash
node backtest/analisar.js                                # analyzes latest batch
node backtest/analisar.js backtest/bets_2026-05-01.json   # analyzes a specific batch
```

## Conventions worth knowing

- **Language**: all UI copy, comments, commit messages, and docs are
  Brazilian Portuguese. Match that when editing existing text.
- **Versioning by comment, not semver**: engine changes are tracked with
  inline `v3.3 — 2026-04-22`-style comments near the constant/function
  that changed, plus the `service-worker.js` `VERSION` string (bump it when
  shipping any change to `index.html` or its assets, so the PWA cache
  invalidates).
- **No destructive backups**: per `.gitignore`, don't create `*.backup_*`
  files — use a git tag (e.g. `v-antes-X-YYYY-MM-DD`) for snapshots instead.
- **Fire-and-forget for non-critical I/O**: Supabase sync calls (MCP bridge)
  and similar background writes are wrapped in try/catch and must never
  surface errors to the user or block rendering — follow that pattern for
  any new "silent sync" feature.
- **`Auditoria Especialista em cantos/` and `Rodada 1005/`** are generated
  output (dossiers, per-matchday HTML/Word reports), largely produced by
  scripts (`_gerar_word.js`, `_gerar_relatorio.js`, `_parser_picks.js`) —
  treat their contents as artifacts, not hand-authored source, unless a
  file is explicitly marked as manually written (e.g. the "gold standard"
  dossiers listed in `00_INDICE_DOSSIES_INDIVIDUAIS.md`).
